## 3. Cassé en prod : constats confirmés par contre-vérification

Le tracker (SPEC §19) marque « fonctionnel » plusieurs choses qui ne le sont pas. La cause commune : le frontend et le backend n'utilisent pas les mêmes noms d'arguments, et le frontend affiche massivement un succès sans lire la réponse (fire-and-forget), donc personne ne voit les échecs.

### C1 : le chat de l'agent ne fonctionne pas ET corrompt Notion à chaque message : CRITIQUE
- Le dock envoie `{prompt}` (sidebar.js:3414) ; le backend lit `body.query` (Code.gs:2400) et répond « query required ». Jamais alignés depuis l'origine (vérifié dans git).
- Le frontend masque l'échec : réponse vide, donc texte de repli « OK, j'ai pris en compte ta demande… » (sidebar.js:3421-3424). L'agent ment.
- Aggravation : le wrapper de logging (Code.gs:3417-3431) s'exécute même sur erreur et écrase la propriété Notion « Agent session ID » de la page Conversation avec un id aléatoire `sb_xxx`. Or cette propriété est la clé de recherche par défaut de TOUTES les fonctions conversation (`ensureConvPage_` 1413, `lookup_conv` 1941, `list_tasks` 1433, `readConvSituation_` 329, `upsert_conv` 1973). Résultat : chaque message du dock rend la page Conversation introuvable, orpheline sa synthèse et ses tâches, et provoque la création d'une page doublon au prochain appel.
- **Fix** : backend, lire `String(body.query || body.prompt || '')` (1 ligne) ; écrire le session id dans une propriété dédiée (« Sidebar session ID »), jamais dans « Agent session ID » ; frontend, afficher `r.error` au lieu du texte de repli.
- **Post-fix** : requêter les pages Conversations dont la clé commence par `sb_` pour réparer les pages corrompues/doublons existants.

### C2 : faux « Document signé » sur la signature Odoo : CRITIQUE
- `callProxy` ne rejette jamais : toute erreur transport (offline, HTTP 500, réponse non-JSON), toute action inconnue et tout token invalide renvoient `{error}` **sans champ `success`** (sidebar.js:218-232, Code.gs:42-43, 100, 103-104).
- Le handler teste `if (r && r.success === false)` (sidebar.js:1366-1372) : `{error}` ne matche pas, branche succès, écran « ✅ Document signé » + footer marqué fait, alors que rien n'a été signé. Avec un token expiré, chaque signature afficherait « signé ».
- **Fix** : inverser le test : `const ok = r && r.success === true;` et afficher `r.error` sinon.

### C3 : la popup de signature peut s'ouvrir sur la mauvaise conversation et signer un vrai document : HAUTE
- `handleConversation` arme un `setTimeout(500)` jamais annulé ni revérifié (sidebar.js:922-927) : un switch de conversation dans la fenêtre ouvre la popup sur la nouvelle conv, avec un nom de document calculé au fire depuis le nouveau sujet.
- Aggravation backend : si le nom ne matche rien et qu'il n'y a qu'une demande de signature en attente, Code.gs:643-645 la signe quand même (« un seul candidat, on le prend »).
- **Fix** : stocker le timer dans `S`, `clearTimeout` en tête de `handleConversation`, revérifier `id === S.conversationId && isOdooSignRequest()` dans le callback.

### C4 : trois actions échouent à 100 % en silence : HAUTE
| Action | Cause | Fix (1-3 lignes chacun) |
|---|---|---|
| Éditer la synthèse (`update_situation`) | front envoie `html`, back lit `text` (Code.gs:550-554) : les corrections manuelles de Benoît ne sont JAMAIS sauvegardées | back : `body.text \|\| body.html` + passage par `htmlToText_` |
| « Ajouter à la veille » (`follow_source`) | payloads sans aucun champ commun (sidebar.js:3307 vs Code.gs:289-302) : bouton passe à « Suivi ✓ » quand même ; en plus `watched:false` déclencherait un AJOUT (pas de chemin retrait) | front : envoyer `source_type`, `contact_page_id/email/name` ; back : ignorer `watched:false` |
| Étoile VIP des participants (`toggle_vip`) | front envoie `{email}`, back exige `page_id` (Code.gs:1273-1286) : no-op silencieux, toast succès quand même | back : fallback `resolveContactPageId_({contact_email})` (helper existant) |

### C5 : les deux boutons Feedback sont cassés à 100 % : HAUTE
- `handleSubmitFeedback_` crée la page avec `parent: {data_source_id}` (Code.gs:497-499) alors que tout le fichier envoie `Notion-Version: 2022-06-28` : ce type de parent n'existe que dans les versions d'API 2025+, donc 400 systématique.
- En plus, le scan nocturne écrit dans la même base avec un schéma incompatible (`Name`/select vs `Titre`/status) : même après correction du parent, au moins un des deux chemins reste cassé.
- **Fix** : `parent: {database_id}` + aligner les deux écritures sur le schéma réel de la base (à vérifier dans Notion) + factoriser en un helper commun.
- À noter aussi : l'édition de la **description** d'une tâche affiche « mise à jour » mais n'est jamais patchée (`handleUpdateTask_` ne gère pas ce champ, Code.gs:532-547).

### C6 : écritures croisées entre conversations (switch rapide) : HAUTE
- `loadConvInstructions` (sidebar.js:320-330) et `saveConv` (355-356) n'ont aucune garde après leurs `await` : en changeant vite de conversation, les instructions de la conv A s'affichent sur la B et le prochain « Sauvegarder » écrit les instructions de B **sur la page Notion de A**. Confirmé de bout en bout.
- `loadTasks` (1942-1953) : même absence de garde + `TaskState` jamais reset au switch : tâches de l'ancienne conv affichées sur la nouvelle ; accepter une tâche proposée dans cette fenêtre la lie à la mauvaise conversation.
- **Fix frontend** : garde `if (convId !== S.conversationId) return;` après chaque await (le pattern existe déjà dans `loadSituation` et `loadContent`) + reset `TaskState` dans `handleConversation`.
- **Fix backend (neutralise les deux à la fois)** : dans `handleUpsertConv_`, vérifier que le `page_id` reçu correspond bien au `missive_conversation_id` avant de patcher (1 garde).

### C7 : une réponse IA mal formée efface la synthèse enregistrée : HAUTE (conditionnelle)
- `parseJsonLoose_` renvoie `{error, raw}` au lieu de `null` (Code.gs:2625-2630) : l'objet passe le garde de `handleRegenSituation_`, summary/bullets/risks deviennent vides, et la persistance écrase les champs Situation par du vide, avec une date « à jour » (la synthèse vide paraît fraîche). Se déclenche dès que le modèle sort du non-JSON (un préambule texte suffit).
- Effet de bord : les gardes `if (!report)` des 8 autres sites d'appel sont du code mort (jamais null).
- **Fix** : `parseJsonLoose_` retourne `null` en échec (+ log du raw) ; ne jamais persister si summary ET bullets sont vides.

### C8 : liens cliquables non filtrés (XSS) : MOYENNE, correctif central
- `esc()` n'échappe pas le scheme des URLs : un lien `javascript:` passe dans les `href`. Vecteur réaliste sur UNE catégorie : les « sources » extraites par l'IA du texte d'un mail externe (un mail attaquant peut faire remonter un lien piégé que l'utilisateur clique). Les 6 autres catégories (PJ Missive, timeline/tâches/meetings Notion, Folk) viennent d'infrastructures de confiance : risque théorique.
- Pas de CSP sur la page pour bloquer l'exécution.
- **Fix** : un helper `safeHref(u)` (n'autorise que `http(s):`, sinon `'#'`) appliqué aux 7 sites de rendu + envisager une meta CSP.

### Constats à portée réduite ou réfutés (contre-vérification)
- « Crash de rendu si tâche sans priorité » : latent seulement, le backend met toujours une priorité par défaut. La seule vraie source (tâches proposées par le dock) est masquée par C1. Fix d'hygiène : `(t.prio || 'P1')` aux 2 endroits manquants.
- « Écran de progression : fuite de timers » : cosmétique (les timers s'arrêtent au retour de la requête). Reste réel : le résultat s'écrit dans `#sheet-content` sans vérifier quel sheet est ouvert. Fix : `stopOdooSignProgress()` dans `closeSheet` + taguer le sheet.
- « Index contacts bloqué à vie après un échec » : RÉFUTÉ (`callProxy` ne throw jamais, l'index retombe sur un tableau vide). Effet réel résiduel : après un échec de `dump_persons`, l'index reste vide toute la session (lookups lents mais fonctionnels).
- « 4tabs boote toujours en mock » : mécaniquement vrai (gating circulaire, ligne 329 vs loader SDK ligne 717) mais fichier legacy non déployé : basse sévérité. À archiver plutôt que corriger.

## 4. UX prod : l'état « aucune conversation » est cassé

Quand aucune conversation n'est sélectionnée (ou qu'elle n'a pas de participant), `body.no-conversation` masque tabs + contenu + footer (sidebar.css:3637-3640), mais l'empty-shell de remplacement n'existe que dans la variante 4tabs : en prod il ne reste que l'en-tête et le dock Agent, le message « Aucun contact identifié » est écrit dans une zone masquée, et le CTA « Lancer le triage » a disparu du produit au passage à 3 onglets. Fix : réintroduire un empty-shell minimal dans index.html, ou ne pas ajouter la classe si le nœud est absent.

## 5. Sécurité (à traiter avec Benoît)

1. **Un seul jeton pour tout, lisible publiquement** : `PROXY_TOKEN` est dans le JS servi par GitHub Pages. Avec ce seul jeton, n'importe qui peut appeler `dump_persons` (export de toute la base contacts : noms, emails, téléphones, instructions), `odoo_sign` (signature juridique réelle au nom de Benoît) et des PATCH Notion sur des pages arbitraires (`add_field_to_notion`, `update_task`, etc. ne vérifient pas l'appartenance de la page à la base attendue). Recommandation : rotation du jeton + second facteur (Script Property non publiée, vérifiée côté GAS) pour les actions sensibles + vérifier `page.parent.database_id` avant tout PATCH.
2. **`setup_config` traité AVANT l'authentification** (Code.gs:38-40) : sur un déploiement frais ou des Script Properties réinitialisées, prise de contrôle possible. Fix : exiger un jeton d'installation même pour cet endpoint.
3. **URLs Zapier en dur dans le source public** (webhook podcast, MCP Folk : Code.gs:26, 1026) : ce sont des URLs porteuses de droits. À déplacer dans Secrets_Proxy.
4. **`SECRETS_PROXY_TOKEN` passé en query string** (Code.gs:2971) : visible dans les logs. À passer en header.
5. **Stack trace renvoyée au client** sur erreur (Code.gs:104). À conditionner à un flag debug.
6. **Injection de prompt** : le contenu d'un mail adverse est injecté tel quel dans le prompt qui produit le verdict juridique OK/ALERT/BLOCK, et dans la synthèse écrite en dur dans Notion/Folk. À encadrer par des délimiteurs + consigne « données, pas instructions ».
7. **Fixtures du mock** (deal ICS, noms, téléphones) embarquées dans le HTML public : vérifier qu'elles sont 100 % fictives.
